# Review Rubric

Use this rubric while writing the daily review.

## A. Evidence hierarchy

1. Deterministic evidence packet
2. Git history and file changes
3. Explicit notes / plan files
4. Session history
5. Memory / recollection

If evidence and recollection conflict, prefer evidence.

## B. What to mark as a mistake

A mistake is worth recording when it caused one of:

- rework
- delay
- wrong implementation path
- repeated failed commands or prompts
- context loss
- decision churn
- debugging thrash
- duplicated explanation

## C. When to create a rule / memory update

Use a rule or memory update when the lesson is:

- stable across tasks
- short enough to remember
- behavior-shaping
- likely to recur

Bad example: "remember this one filename"
Good example: "always collect deterministic evidence before retrospection"

## D. When to create a skill

Create a skill when the work is:

- multi-step
- reusable
- non-obvious
- likely to recur
- improved by a fixed workflow

Bad example: one-off task notes
Good example: summarize hard debugging sessions into root-cause + prevention notes

## E. When to create a script

Create a script when the step is:

- deterministic
- repetitive
- formatting-heavy
- data-collection-heavy
- not improved by freeform prose

Bad example: explaining the same git collection steps daily
Good example: script that gathers git status, same-day commit log, plan files, and history entries

## F. What makes tomorrow easier

The report should make tomorrow faster by giving:

- exact starting state
- exact files to open
- exact commands to rerun
- exact next actions
- explicit traps to avoid
- previously decided boundaries

## G. Publishable insight bar

Only keep social nuggets that are:

- true
- specific
- transferable
- earned from the day's evidence

Delete anything generic, inflated, or motivational.
